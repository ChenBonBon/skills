# 批量文件工作流

仅当用户上传多个图片和/或 PDF 时，使用本参考文档。

当前版本支持：

- `n` 个文件对应 `n` 张财务报表。
- `m` 个文件对应 `n` 张财务报表，且 `m > n`，即多个文件可能组成同一张报表。
- 一个单页 PDF 对应一张报表。
- 一个多页 PDF 对应一张报表。
- 一个多页 PDF 对应多张报表。
- 文件可以是图片或 PDF。

当前版本不支持一张图片包含多个报表单元。如果 OCR 显示一张图片包含多个目标报表，请用户拆分图片，或确认要处理其中哪一张报表。

PDF 可以包含多个报表单元。只能从 `vlm_text` 将 PDF 拆分成报表组；不要使用页面图片、OCR 元数据或文件名作为提取依据。

## 先 OCR 所有文件

1. 在决定分组前，对每个上传的图片/PDF 调用 `ocr`。
2. 立即将 OCR 工具的原始返回对象原样保存到步骤 0 任务输出目录下的 `ocr_results.json`。
3. 不要保存总结、规范化、缩短、重构或占位后的 OCR 内容。
4. 在会话状态中保留每个原始 OCR 响应和 `ocr_results.json` 路径。
5. 后续分组和提取只使用每个响应的 `vlm_text` 字符串数组。
6. 如果任一文件缺失 `vlm_text` 或 `vlm_text` 无效，暂停并请用户重新上传，或请用户手动确认。

## 文件分组与排序

所有 OCR 调用完成后，将文件分成报表单元。每张图片只能属于一个组。若 PDF 的 `vlm_text` 明确包含多个报表单元，一个 PDF 可以产生一个或多个组。

使用 OCR 文本线索推断组成员和组内顺序：

- 明确的表名
- 一个 PDF 内重复出现的明确表名
- 编制单位
- 日期 / 期间
- 单位
- 页码或续页线索
- 表头列
- 首尾科目名称
- 结尾科目，例如 `资产总计`、`负债和所有者权益总计`、`净利润` 或 `现金及现金等价物净增加额`

不要依赖上传顺序。不要只依赖文件名顺序。

规范化后的内部形状为：

```json
{
  "statement_vlm_texts": [
    ["报表1第1段", "报表1第2段"],
    ["报表2第1段"]
  ]
}
```

每个内层数组是一张完整报表的 `vlm_text`。如果一个组包含多个文件，按确认后的组内顺序拼接这些文件的 `vlm_text` 数组。

PDF 处理：

- 单页 PDF 对应一张报表：一个组，包含该 PDF 的 `vlm_text`。
- 多页 PDF 对应一张报表：一个组，包含该 PDF 报表按 OCR 顺序得到的完整 `vlm_text`。
- 多页 PDF 对应多张报表：将该 PDF 的 `vlm_text` 拆分成多个组，每个组对应一个报表单元。使用明确边界，例如 `资产负债表`、`利润表` / `损益表`、`现金流量表`、重复的 `编制单位`、重复的 `日期` / `单位`，以及末行标记。
- 如果拆分存在歧义，请用户确认报表边界后再处理。

## 用户确认

在这个第一版批量文件流程中，始终要求用户确认推断出的分组。

展示简洁表格：

| 组号 | 顺序 | 来源文件 | 识别表名 | 编制单位 | 日期 | 说明 |
|---|---:|---|---|---|---|---|
| 1 | 1 | B.png | 资产负债表 | ... | ... | 表头和资产部分 |
| 1 | 2 | A.png | 资产负债表 | ... | ... | 负债和权益部分 |
| 2 | 1 | C.pdf | 利润表 | ... | ... | PDF 第 1 个报表单元 |
| 3 | 1 | C.pdf | 现金流量表 | ... | ... | PDF 第 2 个报表单元 |

用户可以回复：

- `确认`
- `调整顺序：第1组 B.png,A.png`
- `调整分组：第1组 B.png,A.png；第2组 C.png`
- `调整PDF拆分：第1组 C.pdf 资产负债表；第2组 C.pdf 利润表`
- `取消` 或 `重新上传`

确认后，使用已确认的分组顺序构建 `statement_vlm_texts`。

## 批量 Manifest

对任何多文件上传，或包含多张报表的单个 PDF，使用步骤 0 创建的任务输出目录：

```text
workspace/{username}/result/{original_filename_stem}_{yyyyMMdd_HHmmss}/
```

使用 `scripts/prepare_batch_manifest.py` 在该目录中创建 `batch_manifest.json`。传入已记住的 `task_dir` 作为 `batch_dir`。

所有批量文件都保存在同一个任务目录下。使用每个组在 manifest 中的 `file_prefix` 区分组级 JSON：

```text
group_1_original_validated.json
group_1_subject_mapping.json
group_1_standard_table.json
group_2_original_validated.json
group_2_subject_mapping.json
group_2_standard_table.json
```

`original_filename_stem` 应对本次上传批次保持稳定。优先使用已确认第一个组中的第一个文件。

## 串行处理

按已确认顺序串行处理各组。

对每个组：

1. 用换行符拼接该组的 `vlm_text` 数组。
2. 执行现有单报表工作流，从原始 JSON 映射到标准表 Excel 输出。
3. 如果该组成功，继续下一组。
4. 如果该组需要用户操作，整个批次暂停在该组。

暂停时包含上下文：

```text
当前处理：第 2/5 组
来源文件：B.png, A.png
识别表名：资产负债表
```

用户解决问题后，从暂停组继续。不要重新运行已完成的组。

## 重新上传替换

如果某组失败且用户选择重新上传，用户必须说明要替换哪个报表组。

串行模式下，默认替换当前暂停组，但仍要说明将替换哪个组。未来并行模式中，如果多个组失败且用户没有指定组，询问要替换哪个组。

替换某个组不会重新运行已完成的组。

## 批量成功输出

单组模式保持现有最终成功输出：

```text
标准表校验通过
```

对于多个组，每个组成功后输出简短的上下文成功行：

```text
第 1/3 组：资产负债表 标准表校验通过
```

所有组成功后，输出最终汇总表：

| 组号 | 来源文件 | 表名 | 状态 | 标准表 Excel |
|---|---|---|---|---|
| 1 | B.png,A.png | 资产负债表 | 标准表校验通过 | ... |
| 2 | C.png | 利润表 | 标准表校验通过 | ... |

`标准表 Excel` 列必须填写 `standard_table_to_excel` 返回或复制后的最终 `.xlsx`/`.xls` 路径；不要填写 `standard_table.json` 或任何其他 JSON 路径。

如果批次暂停在失败组，不要输出最终批量完成汇总。
