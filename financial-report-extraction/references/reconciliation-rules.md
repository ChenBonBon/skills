## 请对用户提供的各类格式客户资产负债表进行科目平衡性校验,执行规则如下：

- 自动识别报表内全部资产类合计、负债类合计、股东权益合计.....
- 计算资产负债表中相关勾稽关系是否平衡
- 识别漏填合计数、合计数计算错误、科目错放、期末期初数混淆等常见填报问题
- 最终输出清晰结论：报表是否平衡、差额多少、存在哪些填报漏洞、简要整改方向

## 核心校验规则（必须严格执行，不可违反）

- 【公式1、公式2】校验：资产总计 = 负债和所有者权益总计（直接使用报表合计数对比）
  - 公式1：资产总计（期初）= 负债和所有者权益总计（期初）
  - 公式2：资产总计（期末）= 负债和所有者权益总计（期末）
- 【公式3、公式4】校验：资产总计 = 全部【末级明细资产科目】相加（禁止使用任何合计科目；计算时，如果存在【原值 / 原价】，禁止取【净额】，必须用【原值 / 原价】- 【折价等】计算，若没有【原值 / 原价】，则可以取净值）
  - 公式3：资产总计（期初）= 所有末级资产明细科目相加（期初）
  - 公式4：资产总计（期末）= 所有末级资产明细科目相加（期末）
- 【公式5、公式6】校验：负债和所有者权益总计 = 全部【末级负债明细科目】+ 全部【末级权益明细科目】相加（禁止使用任何合计科目，必须用明细）
  - 公式5：负债和所有者权益总计（期初）= 所有末级负债明细 + 所有末级权益明细（期初）
  - 公式6：负债和所有者权益总计（期末）= 所有末级负债明细 + 所有末级权益明细（期末）
- 如果没有期初值则全部变量按照0处理
- 公式3/4/5/6 绝对不允许使用：流动资产合计、固定资产合计、负债合计、权益合计等包括小的合计项目任计算，要使用子项进行计算，
- 区分取值期间仅在「期间」字段填写，**科目名称本身绝对不能携带期间标识**
- 计算时，如果存在【原值 / 原价】，禁止取【净额】，必须用【原值 / 原价】进行计算
- 计算时，如果不存在【原值 / 原价】的时候可以取净值
- 所有出现的科目名称必须完整，禁止出现丢失前面的序号，丢失括号内容等情况
- 【通过判定零容差】每个公式的左侧结果金额与右侧计算金额必须完全相等，差额必须为 0 或 0.00。任何非零差额都必须判定该公式不通过，包括 0.01、0.02、0.03 等很小差额。
- 不得因为疑似 OCR 识别误差、四舍五入、单位换算、展示精度或金额很小而把非零差额判定为通过。可以在“可能原因”中说明疑似原因，但 `校验是否通过` 必须为 False。
- 总体 `校验是否通过` 只有在所有公式均通过时才可以为 True；任一公式存在非零差额时，总体必须为 False。

## 金额计算与零差额判定

- 勾稽计算必须使用十进制精确计算：从原始 JSON 的金额字符串直接转换为 `Decimal`/等价十进制数后相加减，禁止用二进制浮点数（float/double）作为最终判定依据。
- 金额归一化只允许处理展示格式：去除千分位逗号、空格，将中文逗号转英文逗号后去除，将括号负数转为负号；不得改变金额实际数值。
- 如果计算结果出现 `1e-12`、`0.0000000001`、`-0.0000000001` 等浮点尾差，必须用原始金额字符串重新做十进制计算；不得因为浮点尾差把差额显示为 `0.00` 的公式判定为失败。
- 通过判定只看十进制精确差额：`Decimal` 差额等于 `0` 时通过；`Decimal` 差额不等于 `0` 时失败。不要使用 `abs(diff) < tolerance` 这类浮点容差规则。
- 若十进制精确差额非零，但按两位小数展示会变成 `0.00`，仍判定失败，并在 `计算过程` 或 `可能原因` 中输出足够精度的真实差额，避免展示为 `0.00` 却标记失败。

## 失败后的公式重试（仅一次）

若第一次勾稽失败，先判断失败原因。只有在怀疑失败来自本次勾稽使用的计算公式构造错误时，才允许重试一次；重试不得修改原始 JSON、科目名称、金额、期间或 OCR 内容，只能重新选择计算公式中的科目和运算符。

允许重试的信号：

- 公式中疑似运算符用错，例如应减项被加、应加项被减。
- 公式中疑似重复计算，例如同时使用某个合计科目及其明细，或同一科目被重复加入。
- 公式中疑似遗漏/多计了已有可见科目，并且仅调整公式包含项即可重新计算。
- 公式3/4/5/6 使用了合计科目、结果科目或期间错误的科目，违反了本文件的计算规则。

禁止重试的信号：

- 资产负债表公式1/2（资产总计与负债和所有者权益总计直接对比）失败，且需要改动报表金额才可能通过。
- OCR 文本、科目名称、金额、期间、单位存在缺失、乱码、列错位或疑似识别错误。
- 原始表可见合计数本身与明细不一致，无法只通过替换公式解释为平衡。
- 需要臆造隐藏科目、隐藏金额、四舍五入、单位换算或改写原始金额才能通过。
- 失败原因无法明确区分是公式错误还是源数据/OCR 问题。

重试流程：

1. 保留第一次结果为 `reconciliation_v1`，包括失败公式、计算过程、差额和可能原因。
2. 若满足允许重试信号且不满足禁止重试信号，重新构造失败公式，生成 `reconciliation_v2` 并重新计算一次。
3. `reconciliation_v2` 仍执行零容差；任何非零差额仍失败。
4. 每个报表组最多重试一次；不得继续生成 v3 或循环尝试。
5. 若 `reconciliation_v2` 通过，将 v2 作为最新勾稽结果并继续后续流程；可在内部状态记录“已因公式构造错误完成一次重试”。
6. 若 v2 失败，或判断为禁止重试/无法判断，进入 `references/correction-workflow.md`。失败响应应基于最新失败结果，并在既有“失败原因”或“建议”中说明“公式重试已失败”或“因疑似 OCR/原表不平未执行公式重试”；不要新增第五个响应部分。
